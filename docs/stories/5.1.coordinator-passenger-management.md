# Story 5.1: Coordinator Passenger Management

## Status
Done

## Story
**As a** coordinator,
**I want** to view all children information in a list when clicking the Passengers tab,
**so that** I can see new children added by parents to the system.

## Acceptance Criteria
1. Coordinator can access a Passengers tab in their dashboard
2. Passengers tab displays a list of all children in the system
3. List shows child identification, name, grade, and emergency contact information
4. List updates dynamically when parents add new children
5. Secure access restricted to coordinator role only
6. List is sortable and searchable for efficient navigation

## Dev Notes

### Data Models
**Child Model:**
- childId: String, required, unique - Primary identifier for child
- fullName: String, required - Child's full name
- DOB: Date, required - Date of birth
- grade: String, required - Child's grade level
- allergies: String - Known allergies
- emergencyContact: String, required - Emergency contact information
- parentId: ObjectId ref "Parent", required - Reference to parent profile
[Source: docs/brownfield-architecture.md#Data Models]

**Parent Model:**
- parentId: String, required, unique - Primary identifier for parent
- fullName: String, required - Parent's full name
- phoneNumber: String - Contact number
- email: String - Email address
- address: String - Home address
- password: String - Encrypted password
[Source: docs/brownfield-architecture.md#Data Models]

### API Specifications
**Coordinator Protected Routes:**
- GET /api/children/coordinator/list - Protected endpoint for coordinators to retrieve all children
- Requires coordinator role authentication (JWT token with coordinator claims)
- Response: JSON array of child objects with full details
- Includes related parent information for context
- Status codes: 200 (success), 401 (unauthorized), 500 (server error)
[Source: docs/brownfield-architecture.md#Data Models and APIs]

**Authentication Requirements:**
- Role-based middleware validates coordinator access before API calls
- JWT tokens stored client-side, validated on server requests
- Session management with secure token handling
[Source: docs/brownfield-architecture.md#Integration Points and External Dependencies]
